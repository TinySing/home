# Frontend Agent Rules

本文件用于约束后续 Agent 在前端项目中的目录生成、文件组织和代码编写行为。

目标：

1. 按统一目录和职责逻辑生成代码。
2. 保持现有项目风格一致。
3. 优先复用已有能力，减少无关改动。
4. 生成可落地、可维护、可验证的前端代码。

适用范围：

1. TypeScript 前端项目。
2. React 或类似组件化项目。
3. 存在页面、组件、hooks、service、utils、types、stores 等分层的项目。

---

## 一、生成前必须确认

开始生成或修改代码前，必须先阅读现有项目，确认：

1. 项目目录结构。
2. 页面、组件、hooks、service、utils、types、stores 的现有放置方式。
3. 命名方式和文件命名习惯。
4. 组件拆分方式。
5. 状态管理方式。
6. 接口请求封装方式。
7. 类型定义位置。
8. 是否已有可复用组件、hooks、utils、constants、types。
9. 项目已有验证命令。

边界不清晰时，先说明不确定点，再继续实现。

---

## 二、目录与职责约定

### 1. 基本原则

1. 跟随当前项目已有目录结构，不凭空引入新层级。
2. 优先复用已有目录，不为一次性需求新增复杂目录。
3. 新增目录必须能说明职责和归属。
4. 页面专属代码不放到公共目录。
5. 公共能力不写死在某个页面内。

### 2. 页面

页面负责：

1. 页面结构。
2. 页面交互。
3. 状态组合。
4. 组件编排。
5. 页面级数据接入。

页面中避免：

1. 堆放大量接口请求逻辑。
2. 堆放复杂数据转换。
3. 混写多个无关业务分支。
4. 直接实现可复用的通用逻辑。

### 3. 组件

公共组件：

1. 确认跨页面或跨模块复用后，才放入公共 `components` 目录。
2. 必须通过 props 驱动。
3. 不直接请求业务接口。
4. 不直接依赖页面上下文、页面 store 或具体业务 model。
5. 需要业务能力时，优先通过 props 注入。

页面内组件：

1. 只在当前页面使用的组件，放在当前页面目录下的 `components` 中。
2. 不要提前上提到公共目录。
3. 组件过长或职责过多时，按职责拆分。

### 4. hooks

`hooks` 用于放置可复用逻辑、状态封装和副作用管理。

hooks 负责：

1. 复用状态逻辑。
2. 副作用管理。
3. 事件或行为封装。
4. 页面和组件之间的逻辑隔离。

hooks 中避免：

1. 写组件渲染逻辑。
2. 返回复杂 JSX。
3. 绑定不该复用的页面细节。
4. 把一次性页面逻辑强行抽象成通用 hook。

### 5. service

`service` 用于放置接口相关逻辑。

service 负责：

1. 请求接口。
2. 接口参数处理。
3. 接口返回类型。
4. 接口错误的基础传递或转换。

service 中避免：

1. 写 UI 逻辑。
2. 写组件状态逻辑。
3. 直接依赖页面组件。
4. 混入复杂业务编排。

### 6. utils

`utils` 用于放置无状态纯函数和通用转换逻辑。

utils 适合放：

1. 数据转换。
2. 参数处理。
3. key 生成规则。
4. 状态映射。
5. 权限判断。
6. 文案映射。
7. 通用校验。

utils 中避免：

1. 依赖页面状态。
2. 依赖组件生命周期。
3. 直接请求接口。
4. 放置有副作用的业务流程。

### 7. types

`types` 用于放置类型、枚举和接口结构。

types 负责：

1. 类型定义。
2. 接口结构。
3. 枚举类型。
4. 共享类型声明。

types 中避免：

1. 编写运行时代码。
2. 放置业务函数。
3. 混入组件逻辑。
4. 混入接口请求逻辑。

### 8. stores

`stores` 用于放置状态管理逻辑。

stores 约定：

1. 全局 store 只维护整个项目共享的状态。
2. 页面或模块专属状态优先留在对应页面、模块或 hooks 内。
3. 不要为了方便把局部状态提升成全局状态。
4. 不要让路由层依赖不必要的响应式状态。

## 三、文件组织约定

### 1. import 顺序

import 顺序保持一致：

1. 框架相关。
2. 第三方库。
3. 项目内组件、hooks、utils、services、stores。
4. types。
5. styles。

### 2. 文件内部结构

一个文件内建议按以下顺序组织：

1. 常量。
2. 类型。
3. 主体组件或主体函数。
4. state 和 refs。
5. 派生数据。
6. 方法。
7. effects。
8. render 或 return。

复杂文件可以使用简短分区注释，但注释必须有实际阅读价值。

### 3. 拆分规则

需要拆分的情况：

1. 文件职责明显超过一个模块。
2. JSX 过长且包含多个独立区域。
3. 一个函数内包含多个业务分支。
4. 同一逻辑被多个地方复用。
5. 页面同时承担请求、转换、状态和渲染。

不建议拆分的情况：

1. 只为一次性逻辑制造抽象。
2. 只为显得整洁而增加多层文件。
3. 当前逻辑简单，拆分后阅读成本更高。

## 四、代码编写规范

### 1. 基本原则

1. 严格遵循现有项目风格。
2. 优先选择最小可落地方案。
3. 不修改无关代码。
4. 不顺手重构无关模块。
5. 不重写已有成熟实现。
6. 不添加用户未要求的功能。
7. 不为了未来扩展提前设计复杂抽象。
8. 不无故新增依赖或修改项目配置。

### 2. 优雅编码风格

1. 代码表达应清晰、直接、易读。
2. 禁止使用多层三元运算嵌套。
3. 复杂判断应拆成清晰的变量、函数或条件分支。
4. 简单前置条件优先使用 guard clause。
5. 合理使用解构赋值，减少重复访问深层对象。
6. 解构赋值不能牺牲可读性，层级过深时应拆成中间变量。
7. 布尔判断应尽量简化，避免重复判断和双重否定。
8. 返回值变量名必须表达业务含义，避免使用过于宽泛的命名。
9. 不为了少写几行代码牺牲可读性。

### 3. JSX

1. JSX 中不写复杂逻辑。
2. 复杂判断提前拆到变量、函数或 hooks。
3. 复杂数据转换不要直接写在 render 中。
4. 多业务分支不要堆在一个 render 中。
5. JSX 过长时，优先按职责拆组件。

### 4. TypeScript

1. 类型必须明确。
2. 非必要不要使用 `any`。
3. 接口参数和返回值应有明确类型。
4. 公共函数、hooks、service 应有清晰输入输出类型。
5. 当类型已经明确时，不添加无意义的防御代码。

### 5. 异常状态

真实可能出现的异常状态必须显式处理，例如：

1. 接口失败。
2. 数据为空。
3. 权限不足。
4. 状态不合法。
5. 页面参数缺失。
6. 异步请求被取消或组件已卸载。

异常处理必须清晰可追踪，不允许通过忽略结果掩盖错误。

### 6. 命名

1. 命名必须表达业务含义。
2. 禁止使用 `temp`、`data1`、`handleData` 等模糊命名。
3. 返回值变量名必须表达实际业务含义。
4. 组件名称应表达组件职责，不重复叠加父级语义。
5. 样式类名应带业务语义或业务前缀，避免过于通用。

### 7. 逻辑复用

相同业务表达必须统一复用。

优先复用或抽离：

1. key 生成规则。
2. 状态映射。
3. 参数处理。
4. 权限判断。
5. 文案映射。
6. 数据转换。
7. 通用校验。

优先复用已有 `utils`、`hooks`、`constants`、`types`，不要重复实现已有能力。

### 8. useEffect

1. 避免依赖不稳定函数导致重复执行。
2. 首次执行逻辑必须明确。
3. 不要因为依赖变化导致重复请求。
4. effect 内逻辑过长时，应抽离函数。
5. 异步请求需要处理组件卸载后的状态更新问题。
6. 如果依赖函数会频繁重建，先判断是否真的需要稳定引用。

### 9. 异步调用

1. 需要结果时必须等待结果。
2. 不需要等待结果时，也必须显式处理异常。
3. 禁止用忽略结果的方式掩盖异步错误。
4. 异常处理方式必须清晰可追踪。

### 10. 注释

1. 注释只说明做什么和为什么。
2. 不写无意义描述。
3. 复杂逻辑需要补充关键说明。
4. 新增方法、函数、hook 前可补一行简短注释说明作用。

重点需要注释的场景：

1. 服务端延迟。
2. 兼容逻辑。
3. 状态同步。
4. 历史原因。
5. 风险边界。
6. 特殊交互规则。

---

## 五、最终原则

1. 能复用不原创。
2. 能连接不重造。
3. 能小改不大改。
4. 能明确不模糊。
5. 先理解项目，再定义边界。
6. 先给方案，再小步实现。
7. 解释不清楚，就不要提交。
